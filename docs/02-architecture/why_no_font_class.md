# 为什么没有实现 Font Class 模式？

根据对代码库的分析，推测未实现 Font Class 模式的原因如下：

1.  **文件管理的复杂性**：
    *   **SVG 模式**：只需要下载单个 SVG 文件，或者将 SVG 内容直接写入文件。
    *   **Symbol 模式**：只需要下载一个 JS 文件 (`iconfont.js`)，并将其插入到 HTML 中。
    *   **Font Class 模式**：需要下载多个文件，包括 CSS 文件和字体文件（`.eot`, `.woff`, `.woff2`, `.ttf`, `.svg`）。这涉及到更复杂的文件系统操作和目录结构管理。

2.  **路径引用的处理**：
    *   下载下来的 CSS 文件中，`@font-face` 的 `src` 引用通常是远程 URL（如 `//at.alicdn.com/t/...`）。
    *   如果要本地化，必须下载所有字体文件，并解析 CSS 内容，将远程 URL 替换为本地相对路径。这需要额外的正则解析和字符串替换逻辑。

3.  **现代前端开发的趋势**：
    *   SVG 和 Symbol (SVG Sprite) 提供了更好的渲染质量（不模糊）、支持多色图标、以及更好的 CSS 控制能力。
    *   Font Class 本质上是字体，容易出现锯齿，且只能单色，在现代 Web 开发中逐渐被 SVG 方案取代。

4.  **API 数据结构的依赖**：
    *   插件主要依赖 `/api/project/detail.json` 获取图标数据。SVG 路径数据 (`show_svg`) 直接包含在 API 响应中，易于提取。
    *   Font Class 需要的 CSS 链接可能位于项目详情的不同字段，或者需要单独构建 URL。

尽管如此，Font Class 模式在旧项目维护和特定场景下仍有需求。接下来的更新将尝试支持此模式。
